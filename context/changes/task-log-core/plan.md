# Task Log Core Implementation Plan

## Overview

Build the core task logging feature (roadmap S-01): a `tasks` table scoped to the authenticated user, a Task model, a
controller for creating and listing tasks, and Blade views — replacing the dashboard placeholder with a paginated
chronological task list and an Alpine.js modal for adding new tasks. Covers FR-004 (view tasks in list), FR-005 (scroll
through history), FR-006 (add task with description + optional type tag).

## Current State Analysis

Auth scaffold (F-01) is fully in place. The codebase has:

- `auth` middleware, User model with `HasFactory` + `Notifiable`, Breeze layouts (`app.blade.php`, `guest.blade.php`)
- Alpine.js wired in `resources/js/app.js`; used for navigation and account-deletion modal
- 13 Blade components including `x-modal` (with `x-on:open-modal.window` dispatch pattern), `x-text-input`,
  `x-input-label`, `x-input-error`, `x-primary-button`, `x-secondary-button`
- `ProfileUpdateRequest` establishes the FormRequest pattern
- Dashboard is a closure route (`routes/web.php:11`) returning `dashboard.blade.php` with a placeholder greeting
- SQLite dev database with `users`, `sessions`, `cache`, `jobs` tables — no domain tables yet
- Pest test suite with `tests/Feature/Auth/` covering registration, authentication, password update, and profile

### Key Discoveries:

- The existing `x-modal` component (`resources/views/components/modal.blade.php`) already handles focus trapping, ESC to
  close, backdrop click, and transition animations — reusable for the add-task modal with zero custom JS
- The `delete-user-form.blade.php` partial demonstrates the exact modal dispatch pattern:
  `x-on:click.prevent="$dispatch('open-modal', 'modal-name')"` with a form inside the modal slot
- Dashboard route is a closure in `routes/web.php` — needs to become a controller route to pass task data to the view
- Tailwind CSS v4 via `@tailwindcss/vite` — utility classes work as expected; no config file needed

## Desired End State

After this plan is complete:

1. A logged-in gardener sees their tasks in a paginated chronological list (newest first) on the `/dashboard` route
2. They can click "Add Task" to open a modal, fill in a description (required), date (defaults to today), and optional
   type tag (dropdown with watering/fertilizing/planting presets + custom text), and submit
3. On successful creation, they're redirected back to the list with a flash message confirming the task was added
4. Each task in the list shows its date, description, and type tag (as a plain text label)
5. Pagination links appear below the list (15 tasks per page)
6. All task data is scoped to the authenticated user — users cannot see each other's tasks
7. Pest feature tests verify the full request lifecycle; a model unit test covers relationships and scoping

**Verification**: Register a new user, add 3+ tasks (one with each preset type, one with custom type, one with no type),
verify they appear in the list newest-first, verify pagination by seeding 20+ tasks, verify another user cannot see the
first user's tasks.

## What We're NOT Doing

- **FR-009/FR-010 (AI recall)** — that's S-02 `ai-recall-loop`

- **FR-012 (week-view calendar)** — nice-to-have, deferred per roadmap
- **API routes** — this slice is Blade-only; no `routes/api.php` needed

## Implementation Approach

Standard Laravel resource pattern: migration → model → form request → controller → routes → Blade views → tests. The
add-task modal reuses the existing Breeze `x-modal` component and Alpine.js dispatch pattern established in the profile
deletion flow. The dashboard closure route is replaced by a `TaskController@index` method that queries paginated tasks
for the authenticated user.

## Phase 1: Data Layer

### Overview

Create the `tasks` table migration, Task Eloquent model with user relationship and scoping, and a TaskFactory for test
data generation.

### Changes Required:

#### 1. Tasks migration

**File**: `database/migrations/xxxx_xx_xx_xxxxxx_create_tasks_table.php` (generated by `php artisan make:migration`)

**Intent**: Create the `tasks` table with columns for user ownership, description, date, and optional type tag. The
`user_id` foreign key enforces referential integrity and enables efficient per-user queries.

**Contract**: Table `tasks` with columns: `id` (primary), `user_id` (foreign key → `users.id`, cascading delete,
indexed), `description` (text, not null), `task_date` (date, not null, indexed), `type` (string 100, nullable),
`created_at`/`updated_at` (timestamps). Composite index on `(user_id, task_date)` for the chronological list query.

#### 2. Task model

**File**: `app/Models/Task.php` (generated by `php artisan make:model`)

**Intent**: Eloquent model representing a garden task. Belongs to a User; provides a scope for filtering by
authenticated user.

**Contract**: `$fillable = ['description', 'task_date', 'type']`. `$casts = ['task_date' => 'date']`. Relationship:
`user(): BelongsTo`. Scope: `scopeForUser(Builder $query, User $user)` filtering by `user_id`.

#### 3. User → tasks relationship

**File**: `app/Models/User.php`

**Intent**: Add the inverse `hasMany` relationship so tasks can be accessed from the user model.

**Contract**: New method `tasks(): HasMany` returning `$this->hasMany(Task::class)`.

#### 4. Task factory

**File**: `database/factories/TaskFactory.php` (generated by `php artisan make:factory`)

**Intent**: Factory for generating test tasks with realistic garden data.

**Contract**: `definition()` returns `user_id` (from `User::factory()`), `description` (fake sentence), `task_date` (
random date within last 90 days), `type` (random from `['watering', 'fertilizing', 'planting', null]`).

### Success Criteria:

#### Automated Verification:

- Migration applies cleanly: `php artisan migrate`
- Migration rollback works: `php artisan migrate:rollback`
- `php artisan tinker` — `Task::factory()->create()` produces a valid record
- `composer test` — existing test suite still passes (no regressions)

#### Manual Verification:

- Inspect `database/database.sqlite` — `tasks` table exists with correct columns and indices

## Phase 2: Backend Logic

### Overview

Create the TaskController with index (list) and store (create) actions, a StoreTaskRequest for validation, and wire the
routes with auth middleware.

### Changes Required:

#### 1. StoreTaskRequest

**File**: `app/Http/Requests/StoreTaskRequest.php` (generated by `php artisan make:request`)

**Intent**: Validate incoming task creation data. Description is required (max 500 chars), date is required and must not
be in the future, type tag is optional (max 100 chars).

**Contract**: `authorize()` returns `true` (auth middleware handles access). Rules: `description` →
`['required', 'string', 'max:500']`, `task_date` → `['required', 'date', 'before_or_equal:today']`, `type` →
`['nullable', 'string', 'max:100']`.

#### 2. TaskController

**File**: `app/Http/Controllers/TaskController.php` (generated by `php artisan make:controller`)

**Intent**: Handle listing and creating tasks for the authenticated user.

**Contract**: Two methods:

- `index()` — queries `auth()->user()->tasks()->latest('task_date')->paginate(15)` and returns the `dashboard` view with
  the paginated tasks collection.
- `store(StoreTaskRequest $request)` — creates a task owned by `auth()->user()` from validated data, flashes a success
  message, redirects back to dashboard.

#### 3. Route wiring

**File**: `routes/web.php`

**Intent**: Replace the dashboard closure route with `TaskController@index` and add a POST route for task creation. Both
under `auth` middleware.

**Contract**: `Route::get('/dashboard', [TaskController::class, 'index'])->middleware('auth')->name('dashboard')`.
`Route::post('/tasks', [TaskController::class, 'store'])->middleware('auth')->name('tasks.store')`. Remove the existing
closure route for `/dashboard`.

### Success Criteria:

#### Automated Verification:

- `php artisan route:list` shows `/dashboard` pointing to `TaskController@index` and `/tasks` POST pointing to
  `TaskController@store`
- `composer test` — existing test suite still passes
- No route name collisions: `route('dashboard')` and `route('tasks.store')` resolve

#### Manual Verification:

- Visit `/dashboard` while logged in — page loads without errors (view will be updated in Phase 3; at this point it may
  show the old placeholder or break if `$tasks` is missing — acceptable, Phase 3 fixes it)

**Implementation Note**: After completing this phase and all automated verification passes, pause here for manual
confirmation from the human that the manual testing was successful before proceeding to the next phase.

## Phase 3: Frontend Views

### Overview

Replace the dashboard placeholder with the task list view, add the "Add Task" modal using the existing `x-modal`
component, implement flash message display, and wire pagination.

### Changes Required:

#### 1. Dashboard view overhaul

**File**: `resources/views/dashboard.blade.php`

**Intent**: Replace the placeholder greeting with the chronological task list and an "Add Task" button. The view
receives a `$tasks` paginated collection from `TaskController@index`.

**Contract**: Uses `x-app-layout`. Header slot shows "My Garden Log". Main content area contains:

- Flash message banner (conditionally shown via `session('success')`)
- "Add Task" button dispatching `open-modal` event for `'add-task'`
- Task list: iterates `$tasks`, each row showing `task_date` (formatted), `description`, and `type` (as a plain text
  badge when present)
- Empty state message when no tasks exist
- `$tasks->links()` for pagination

#### 2. Add-task modal

**File**: `resources/views/tasks/partials/add-task-form.blade.php` (new partial)

**Intent**: A form inside the `x-modal` component for creating a new task. Follows the same pattern as
`delete-user-form.blade.php`.

**Contract**: `x-modal` with `name="add-task"` and `focusable`. Contains a POST form to `route('tasks.store')` with:

- `description` textarea (required)
- `task_date` date input (defaults to today via `value="{{ old('task_date', now()->format('Y-m-d')) }}"`)
- `type` select dropdown: empty option ("No type"), watering, fertilizing, planting, "Other (custom)" — selecting "
  Other" reveals a text input (Alpine.js `x-show` toggle)
- Validation error display via `x-input-error` for each field
- Cancel button (`x-on:click="$dispatch('close')"`) and Submit button

#### 3. Include modal partial in dashboard

**File**: `resources/views/dashboard.blade.php`

**Intent**: Include the add-task modal partial so it renders on the dashboard page.

**Contract**: `@include('tasks.partials.add-task-form')` placed after the main content section.

- `npm run build` — Vite build succeeds with no errors

## Testing Strategy

### Unit Tests:

- Task model belongs-to User relationship
- User has-many Tasks relationship
- `task_date` Carbon cast
- `scopeForUser` scope filters correctly

### Integration Tests (Feature):

- Full request lifecycle: create task → see it in list
- Auth guard on both routes
- Validation rules enforce description required, date not-future, type optional
- Pagination at 15 per page
- User scoping — cross-user isolation

### Manual Testing Steps:

1. Register a new user, verify empty task list with call-to-action
2. Add a task with "watering" type → appears in list with badge
3. Add a task with custom type "pruning" → appears with custom badge text
4. Add a task with no type → appears without badge
5. Backdate a task to last week → appears in correct chronological position
6. Seed 20+ tasks → pagination links appear and work
7. Log in as a different user → see only their own tasks (empty if new)

## Performance Considerations

- Composite index on `(user_id, task_date)` ensures the chronological list query is indexed
- Pagination at 15 items limits query + render cost per page load
- No N+1 concern — tasks are loaded directly, not through nested relationships in the list view

## Migration Notes

Greenfield project — no existing data to migrate. The `tasks` table is new and has no dependencies beyond the `users`
table foreign key.
